# 概念去重判定规范

概念是自然语言定义的抽象思想，无法做算法化比对。因此概念去重依赖 Agent 语义判断，而非硬阈值流水线。

---

## 0. 问题定义

一个新概念 `C_new` 准备创建时，需判断与已有概念 `C_existing` 的关系：

| 关系 | 定义 | 处置 |
|:---|:---|:---|
| **精确重名** | 文件名完全一致（蛇形命名法） | ⛔ 拒绝新建，改为增量追加 |
| **同义重复** | 文件名不同但定义同一事物（如 `knowledge_distillation` vs `model_distillation`） | ⛔ 选择规范名新建，另一个做重定向或合并 |
| **子概念** | C_new 是 C_existing 的一个专门化（如 `chain_of_thought` 是 `prompt_engineering` 的下位方法） | ✅ 独立建页，通过 `related_nodes` 双向链接 |
| **近义概念** | 定义有重叠但边界不同（如 `transfer_learning` vs `domain_adaptation`） | ⚠️ 谨慎建页，明确边界差异，互设链接 |
| **交叉领域** | 属于不同分类维度但有交叉引用 | ✅ 独立建页，在索引中放在多个分类下 |
| **全新概念** | 与库中所有概念无重叠 | ✅ 正常建页 |

## 1. Agent 执行流程

概念去重目前是**弱信号 + Agent 判断**模式。Agent 在创建概念前必须执行以下检查：

```
准备创建新概念 C_new
  │
  ▼
┌─────────────────────────────────┐
│ 第1步：索引级碰撞检查             │
│ 读 wiki/concepts/index.md        │
│ 检查文件名在已有条目中是否存在    │
└─────────────────────────────────┘
  │
  ├── 文件名命中 → ⛔ 精确重名。增量追加到已有页面，不建新页。
  │
  ▼ 未命中
┌─────────────────────────────────┐
│ 第2步：lookup_key 碰撞检查        │
│ 搜索所有已有概念页的 frontmatter  │
│ 中 lookup_keys 字段               │
│ 是否有与 C_new.lookup_keys 重叠   │
└─────────────────────────────────┘
  │
  ├── 大量重叠（>50%） → ⚠️ 读已有页 ABSTRACT，判断是否同义重复
  │
  ▼ 无重叠或少量重叠
┌─────────────────────────────────┐
│ 第3步：语义边界判断               │
│ 读可能与 C_new 邻近的概念页       │
│ ABSTRACT + 结构化数据定义字段     │
│ 判断 C_new 是否是该概念的下位/    │
│ 平行/重叠/新领域                  │
└─────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────┐
│ 第4步：处置决策                   │
│ 按 §0 关系表选择对应的处置方式    │
│ 建新页 OR 增量追加 OR 重定向      │
└─────────────────────────────────┘
```

### 各步骤详解

#### 第1步：索引级碰撞检查

读取 `wiki/concepts/index.md`，提取所有 `[[概念名]]` 作为已有概念集合：

```bash
# Agent 执行的等价命令
grep -oP '\[\[[a-z_]+\]\]' wiki/concepts/index.md | sort -u
```

若文件名完全一致 → 拒绝新建，进入"增量追加"模式（在已有页末尾追加 `## 补充` 章节）。

#### 第2步：lookup_key 碰撞检查

每个概念页的 frontmatter 中 `lookup_keys` 字段存放中英文搜索关键词。Agent 读取所有已有概念的 lookup_keys，与 C_new 的计划 lookup_keys 做交集比较：

```
已有 lookup_keys: [prompt tuning, prompt engineering, 提示调优, soft prompt]
C_new lookup_keys: [prompt tuning, prefix tuning, 前缀调优, continuous prompt]
重叠: ["prompt tuning"] → 1/4 = 25%，低重叠，进入第3步
```

**阈值判断**：
- 重叠 > 50%：高度疑似同义重复，必须仔细读已有页面后才能决定
- 重叠 20-50%：可能相关但不同，进入第3步
- 重叠 < 20%：大概率新概念，但第3步仍不可跳过

#### 第3步：语义边界判断

这是**无法算法化的核心环节**。Agent 需要读候选邻近概念页的：

1. **ABSTRACT** — 一句话核心结论
2. **结构化数据 → 定义** — 精炼定义
3. **related_nodes** — 已有关系网络
4. **溯源 → confidence** — 可靠性

然后做语义判断：C_new 和 C_existing 的关系落入了 §0 关系表的哪一行？

#### 第4步：处置决策

按 §0 表执行。若决定建新页，必须同步更新：
- `wiki/concepts/index.md`（追加条目）
- `wiki/index.md`（追加条目）
- 相关已有概念页的 `related_nodes`（添加双向链接）
- `wiki/changelog/log.{date}.md`（记录操作）

## 2. 实际案例：fine_tuning vs prompt_tuning 去重

这是概念去重流程的一个典型案例，展示了如何区分"近义概念"和"同义重复"。

### 判定过程

```
C_existing = fine_tuning (全模型参数微调)
C_new      = prompt_tuning (仅调优提示嵌入)

第1步：文件名 → fine_tuning vs prompt_tuning → 不重复 ✓
第2步：lookup_keys →
  fine_tuning:   [fine tuning, 微调, parameter update, full finetune]
  prompt_tuning: [prompt tuning, 提示调优, soft prompt, prefix tuning]
  重叠: ["微调"] = 1/8 = 12.5% → 低重叠 ✓
第3步：语义边界 →
  fine_tuning:    更新全部模型参数以适应下游任务（通用训练范式）
  prompt_tuning:  冻结模型参数，仅调优输入嵌入（参数高效方法）
  结论：近义概念关系（共享"微调"语义范畴但机制不同），非重复 ✓
第4步：处置 →
  ✅ 独立建 prompt_tuning.md
  ✅ fine_tuning.md 补充指向 prompt_tuning.md
  ✅ 均在 index.md 的"训练方法"分类下列出
```

### 为什么不能合并？

如果把 prompt_tuning 塞进 `fine_tuning.md`：
- 混淆了两种本质不同的机制：全参数更新 vs 冻结+嵌入调优
- 其他参数高效方法（如 `lora`、`adapter`）无法形成平行对照
- 读者无法快速对比不同微调范式的适用场景

## 3. 为什么概念不能做算法化去重？

概念去重与代码/公式去重的本质差异在于表达方式：

```
公式/代码:  字符串 → AST 算子树 → 编辑距离 → 可计算！
概念:      "涌现能力" vs "突现行为" → 同一概念，不同叫法 → 算法无解！
```

两个概念可以用完全不同的词组描述同一事物。反之，同名的概念在不同领域可能有不同定义（如 `attention` 在 NLP 和 CV 中的差异）。这些判断只能靠领域知识。

## 4. 质量检查中的概念去重

`wiki-lint` 技能中规定 Agent 需定期执行冲突检测——检查 `wiki/concepts/` 下是否存在对同一术语的不同定义（见 AGENTS.md §2 编辑规则中的技能参照）。

Agent 执行的 Lint 动作：

```bash
# 1. 提取所有概念页的 lookup_keys 并找重叠
# 2. 对每个重叠对，读定义区做语义比对
# 3. 若发现冲突定义 → 在日志中记录，必要时做增量合并或标记
```

当前已知的无冲突状态（以 `fine_tuning` vs `prompt_tuning` 为例）：
- `fine_tuning` 定义的是"全参数更新以适应下游任务"
- `prompt_tuning` 定义的是"冻结模型、仅调优输入嵌入"
- `lora` 定义的是"低秩适配，另一类参数高效方法"
- 三者无定义冲突，但在 lookup_keys 上有重叠（共享`微调`关键词）——这是**预期的交叉引用**，不是重复

## 5. 未来改进方向

| 方向 | 描述 | 可行性 |
|:---|:---|:---|
| **概念签名** | 为每个概念定义结构化的"5维标签"，实现粗略的自动比对 | 🟡 中 |
| **本体关系图** | 构建概念间的正式本体关系图（is-a / part-of / quantified-by），自动发现缺失或重复节点 | 🟡 中 |
| **向量嵌入比对** | 用 embedding 模型将概念定义编码为向量，余弦相似度作为去重信号 | 🟢 高 |
| **周期性自动扫描脚本** | 定时运行，检查 lookup_keys 重叠、文件名近似（编辑距离）、ABSTRACT 向量相似度，输出候选冲突报告 | 🟢 高 |

当前阶段（Phase 0）仍是 Agent 手工判定。若概念数量增长到 50+ 或多人协作写入，建议优先实现"向量嵌入比对"作为弱信号辅助。
