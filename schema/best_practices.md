# Agent 最佳实践

本文档记录 Agent 在维护 AI 知识库 wiki 过程中积累的实践经验，与 `AGENTS.md` 互补——AGENTS.md 定义**协议**（做什么），本页记录**经验**（怎么做更好、已知的坑）。

## 1. 页面融合优于独立

**教训**: 发现多页内容重叠时，主动融合，不为每个子话题都建独立页。

本会话中 Transformer 架构的三种变体（Encoder-only、Decoder-only、Encoder-Decoder）最初分散在三篇不同的 source 页中讨论，每页都重复介绍自注意力机制。融合为 `transformer_architecture` 概念页后，三篇 source 页只保留架构差异对比，公共机制集中定义，消除了重复。

**原则**: 一个概念只在一个地方定义。新增内容时先问"是否已有页面能承载？"再问"是否需要新建？"

---

## 2. 受控词表必须从实践中生长

**教训**: 初始 Tag 受控词表来自经验设计，但实际 wiki 使用中发现了大量未覆盖的 tag——规范形同虚设。

实际修正过程揭示了正确的做法：

1. 先用 `grep` 提取所有实际使用的 tag → 按频次排序
2. 分类（方法论/理论学科/技术方法/特殊）
3. 每类保留频次 ≥ 3 的 tag
4. 将完整的词表维护在 `schema/tags.md`，协议文件只保留规则和一句引用

**原则**: 受控词表是实际使用的**映射**，不能凭空设计。先有使用，再有分类。

---

## 3. 概念页的"三层验证"

创建一个概念页后，不能只更新索引就算完。必须通过三层验证：

| 验证层 | 检查方法 | 本会话的补救案例 |
|:---|:---|:---|
| **嵌入性** | 该概念是否被至少一个 synthesis 页的"相关页面"引用？ | `self_attention` 创建后未加入 `attention_mechanism` 的相关页面——后补 |
| **交叉性** | 该概念与相关概念之间是否有双向链接？ | `attention_mechanism` 创建时链接了 `transformer`，但反过来没有——需补全 |
| **演示性** | 是否有书面案例展示该概念的使用场景？ | 6 个注意力变体概念各自有定义，但没有"怎么组合"的演示 → 催生了 `attention_architecture_survey` |

**原则**: 概念页的价值不在于存在，在于被引用、被展示、被连接。三者缺一即"孤岛概念"。

## 4. 装配矩阵优于线性叙述

**教训**: `attention_architecture_survey` 初版用线性叙述逐一介绍 6 种注意力变体，结果读者难以横向对比各变体的适用场景和计算复杂度。

重修采用**装配矩阵**（注意力变体 × 任务类型 × 复杂度）：

```
变体             长序列    机器翻译    图像    计算复杂度
self_attention    ✅       ✅        ✅      O(n²)
cross_attention   ❌       ✅        ❌      O(n×m)
linear_attention  ✅       ❌        ✅      O(n)
 ...
```

矩阵的强制对齐属性暴露了线性叙述无法暴露的问题——每个单元格都必须填，空单元格一目了然。

**原则**: 当需要展示 N 个实体在 M 个维度上的关系时（N>3 且 M>2），用矩阵/表格替代叙述。

## 5. 从具体问题推导通用概念

**教训**: "涌现临界点"这个概念不是预先设计的——是从"为什么小模型没有推理能力"推导出来的。

推导路径：
1. 观察到：模型在参数量超过某个阈值后突然出现链式推理能力
2. 追问：为什么小模型没有？（答案：参数量未跨越临界点）
3. 对照：不同能力的涌现临界点不同——算术在 6B 涌现，推理在 60B 涌现（答案：任务复杂度决定临界阈值）
4. 提炼：模型能力增长不是平滑曲线，而是阶段函数 → "涌现临界点"

**原则**: wiki 的概念页不应该只记录"已知概念"——最有价值的概念是从数据/现象中**提炼**出来的。遇到反直觉的实证现象，追问"为什么"直到出现可泛化的解释。

## 6. Source / Concept / Entity 铁三角

**教训**: 每篇原始论文（`raw/`）摄入时，默认检查三角：

| 页面 | 问题 | "Attention Is All You Need" 的处理 |
|:---|:---|:---|
| Source | 有没有源文件页？ | ✅ 创建 `vaswani_2017_attention` |
| Entity | 作者有没有实体页？ | ✅ 创建 `ashish_vaswani` |
| Concept | 论文引入的新概念有没有定义页？ | ✅ 创建 `transformer_architecture`、`attention_mechanism` |

**原则**: 摄入论文后，缺一不可。如果有两个缺失，优先创建 source 和 concept，entity 次之。

## 7. 合成页的"碎片收集"模式

**教训**: `attention_mechanism` 的创建暴露了 wiki 的一个典型增长模式——同一个主题的碎片分散在 6 个页面中：

| 碎片 | 所在页面 |
|:---|:---|
| 自注意力定义 | `transformer_architecture` |
| 跨注意力应用 | 机器翻译 source 页 |
| 多头注意力机制 | `multi_head_attention` 概念页 |
| 线性注意力变体 | `linear_transformer` source 页 |
| 因果注意力 | `gpt_architecture` 概念页 |
| 注意力可视化 | 各篇论文 source 页的散落记录 |

**原则**: 当一个主题在 ≥3 个页面中被分散讨论时，主动创建合成页统一叙事。合成页不是摘抄——是找到一个串起所有碎片的统一框架。`attention_mechanism` 的统一框架是"注意力 = 查询(Q)与键值(KV)的交互函数"。

## 8. 日志的统一结构

本会话的日志条目遵循以下结构，经验证有效：

```markdown
### [日期] [操作类型] 简述
- **动机**: 为什么做
- **操作**: 做了什么
- **新建/修改**: 文件列表 + 核心内容
- **交叉引用**: 哪些页面被联动更新
- **索引更新**: 哪些 index 被修改
```

**原则**: 每条日志至少覆盖动机和操作。日志不是给 AI 看的——是给三天后的自己看的（"我当时为什么建这个页面？"）。

## 9. Commit 粒度：逻辑完整单元

本会话的 commit 边界划分原则：

| 粒度 | 本会话案例 |
|:---|:---|
| 结构重构 | AGENTS.md 结构精简 |
| 概念组 | 注意力变体概念 × 6 个 → 一 commit |
| 合成页 | `attention_mechanism` + `attention_architecture_survey` 各自独立 commit |
| 重大重修 | `attention_architecture_survey` 装配矩阵重修 → 单独 commit |
| 概念补充 | `emergence_threshold` → 单独 commit |

**原则**: 每 commit 一个逻辑完整的工作单元。不对应文件数——6 个概念一组的 commit 和 1 个合成页的 commit 都是合理的单元。避免"今天的全部"巨型 commit。

## 10. "概念链"的建页节奏

从本会话的建页顺序可以抽象出一个通用模式：

```
现象 → 追问 → 概念 → 合成 → 验证
```

本会话的具体路径：

```
现象: Transformer 在不同任务上表现差异显著
  → 追问: 核心机制是什么？
  → 概念: attention_mechanism (注意力机制)
    → 追问: 有哪些变体？
    → 概念 × 6: self/cross/multi-head/linear/causal/sparse attention
      → 追问: 怎么选型？
      → 合成: attention_architecture_survey (装配矩阵)
        → 追问: 不同变体在不同任务上的效果规律？
        → 合成: attention_mechanism_comparison (实证对比)
          → 追问: 效果差异的底层数学原理？
          → 概念: qkv_projection
            → 验证: 涌现临界点概念（从"为什么小模型没有推理能力"推导）
```

**原则**: 每一次追问的答案如果 ≥ 3 段文字，它就值得一个独立页面。如果 < 3 段，嵌入到父页面中。

## 11. Edit 工具：空 oldString 会覆盖全文

**教训**: `attention_variants.md` 用 `write` 创建后，想用 `edit` 追加内容——但 `oldString` 传了空值，结果编辑工具执行全文替换，文件只剩下追加的那部分。原始内容的分类表格全部丢失，122 行 → 81 行。

**根因**: `edit` 工具在 `oldString` 为空 `""` 时的行为是全文件内容替换，而非追加或报错。这个行为与直觉相反。

**正确做法**:
- **追加内容** → 匹配文件最后一行作为 `oldString`，把 `newString` 设为 `原最后一行 + 新内容`
- **不确定匹配文本** → 用 `read` 获取全文 + `write` 重写，不用 `edit`
- **永远不要传空 `oldString`** 给 edit 工具

**恢复代价**: 必须用 `write` 重新写出整份文件的三级别完整内容。

## 12. wiki 导航设计：问题驱动优于分类学驱动

**背景**: 用户提出将 synthesis 页按"纵向串联(演化史) / 横向关联(交叉对比) / 行动指南 / 路线图"四象限分类。经批判性分析，该分类存在三个问题：
① 好页面天然跨象限，强行归位丢失维度；② 分类不回答任何实际问题；③ 概念页的"原子化"在实践上是伪命题。

**确定的原则**:
- **一级导航 = "我想做什么"**，而非"这是什么类型"。synthesis/index.md 顶部应为问题→答案映射，底部保留按类型的深度浏览
- **related_nodes 应有方向性**：关键概念链（如 Attention → Transformer → GPT）的页面底部需要 "→ 下一步: [[xxx]]" 导航，而非平铺所有相关链接
- **批量摄入后需要发现性入口**：42 个新页面涌入后，需创建"阅读入门"页按"先读 → 再读"排列，解决选择困难

**对 Agent 的要求**: 新创建 synthesis 页时，优先考虑"读者拿着这页要解决什么问题"而非"这页属于哪个象限"。更新 index.md 时追加到对应"我想..."条目下。
