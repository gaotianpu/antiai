## 核心流程

当用户说"整理工作流"或"整理关于xxx的工作流"时，执行以下步骤：

### 1. 回顾会话
通读当前对话，识别主干任务（不是每个工具调用，而是用户发起的一次完整意图）。

### 2. 划分轮次
按"用户发起 → 产出物落成"分轮，每轮一个标题。

判断规则：
- **算一轮**：用户发起一次完整意图（如"规整X文档"）→ 产出物落地 = 一轮
- **不算一轮**：用户的追问（"确认"、"好"）属于轮内步骤，不单开
- **轮次边界**：同一用户消息包含多个独立意图 → 按意图拆多轮

### 3. 提取弯路
找出"做了又撤销"、"先错后改"、"重复讨论"的节点，分析根因和应对。

判断规则：
- **算弯路**：(1) 做了又撤销；(2) 两轮讨论同一主题但没推进（语义重叠）；(3) 顺序错误导致返工；(4) 文档和代码不同步导致的批量修复
- **不算弯路**：正常迭代（用户追加需求）、一次性正确完成的任务、用户显式说"先搁置"的事项

### 4. 提炼理想流程
基于弯路，反向推导正确顺序。

### 5. 填写输出清单
只列本轮会话产出的文件，不改动的不列。

## 高频弯路预检清单

整理工作流时，对照以下 3 条根因检查是否触发。触发即标记为弯路。

| # | 根因 | 识别信号 | Agent 动作 |
|---|------|---------|-----------|
| P1 | 内容重复 | 同一概念在文档出现 ≥2 次且描述不一致 | 写完文档后 grep 关键概念，提出"只在一处详写，其余引用 `（见 §X）`" |
| P2 | 设计/结构未确认就动手 | 参数/章节加了又删；用户说"A"后直接实现无确认 | 听到新增/删除指令时追问"核心还是可选？"；写文档先确认骨架 |
| P3 | 文档与代码更新不同步 | 代码已改多轮文档仍引用旧名/旧路径 | 每次改代码后主动问"关联文档要不要同步？"；改路径后 grep 全项目确认 |

## 文档模板

输出文件按项目约定命名（如 `logs/workflow.{topic}.md`），`{topic}` 取用户问题中最简短的关键词组合。

```markdown
<!-- 元数据头 -->
# {主题} 工作流

> 记录本轮会话从起点到全部产出的完整过程。

## 一、工作流全景

用 ASCII 时间线展示各轮次：
```
第1轮  xxx  ── 动作描述
第2轮  xxx  ── 动作描述
...
```

## 二、各轮详情

每轮：
### 第 N 轮：标题
**起点**：用户的一句话意图。
| 动作 | 产出 |
|------|------|
| ... | ... |
> **教训**：一句话总结本轮学到的方法论。

## 三、弯路

| # | 弯路 | 根因 | 应对 |
|---|------|------|------|
| 1 | 做了什么 → 撤销 | 为什么出错 | 下次怎么做 |

## 四、理想流程

```
步骤 1 → 步骤 2 → ... → 步骤 N
```
对比表格：偏差处标注理想 vs 实际

## 五、输出清单

| 类型 | 文件 | 说明 |
|------|------|------|
| 代码 | ... | ... |
| 文档 | ... | ... |
```
