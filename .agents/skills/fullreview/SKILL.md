## 核心流程
文档初稿写完后，回头做一次全局审查，按以下 5 步逐项执行：

1. **逐段扫描重复** — 同一概念在多个章节出现 → 合并，只保留首次出现的详细版，后续用引用 `（见 §X）`
2. **检查前后一致性** — 数字/公式/命名是否统一；表结构和代码示例是否与实现匹配
3. **概念归位** — 已实现的功能不放"未来可扩展"/"TODO"；每个概念归属到其自然章节
4. **认知流检查** — 读者能否不跳转就理解？章节顺序(供参考，非必须遵守)：概述 → 使用方式 → 核心概念 → 实现细节 → 扩展
5. **删除过时内容** — 已移除的参数（CLI 别名）、与实现不符的伪代码/示例、已被替代的旧方案

## 参考案例
以下 7 类问题来自 `ohlcv_data_checker.md` 和 `workflow_check_ohlcv_data.md` 的规整实操，每类给出反例和修法。

### 问题 1：重复 / 冗余
同一概念、说明、公式或约束在多个章节出现。

**反例 A（字面重复）**：
- **描述**：`--opt=VALUE` 语法说明在"使用场景"章节和"参数解析"章节各出现一次，内容相同。
- **修法**：只保留首次出现的详细版，后续用引用 `（见 §使用方式）`，或直接删除后出现的版本。

**反例 B（语义冗余）**：
- **描述**: "时间检查逻辑"一段伪代码与"检查项"表格语义完全重复——表格已逐项说明了排序检查、间隔检查、跳空检查、重复检查、期望行数。
- **修法**：确认哪个版本更好（表格 > 伪代码），删除另一个。如果两处各有价值，合并为一处。

### 问题 2：前后不一致（与实现不同步）
文档中的数据结构、流程、示例与代码实现不一致。

**反例 A（数据结构过时）**：
- **描述**：`CheckResult` 定义缺少已实现的 `gap_details` 字段。
- **修法**：对照源码中的 dataclass / 接口定义，逐字段核对后补全。

**反例 B（流程缺失）**：
- **描述**：核心流程图缺少 `sort_values`、`write_output_files` 步骤。
- **修法**：跑一遍代码的执行路径，将遗漏的步骤补进流程图。

**反例 C（示例不同步）**：
- **描述**：终端输出示例是旧的表格格式，与实际输出列不对齐。
- **修法**：复制一次实际运行输出替换文档中的旧示例。

**反例 D（术语不一致）**：
- **描述**："对照映射"在不同章节分别叫"对照映射"/"对照映射阶段"/"doc vs code diff"，读者可能误以为是不同概念。
- **修法**：选定一个主名称全文统一，备用叫法只在首次定义处标注 `（也称 xxx）`。


### 问题 3：概念错位
功能的文档位置与其实际状态不符——已实现的功能放在"未来可扩展"/"TODO"，或碎片章节散落各处。

**反例 A（已实现归入扩展）**：
- **描述**：`gap.csv` / `last_ts.csv` 已实现并自动输出，却放在"未来可扩展"章节。
- **修法**：将已实现项上浮到对应的正式章节（"输出"）。

**反例 B（碎片章节）**：
- **描述**："异常处理"作为独立子章节散落在实现章节末尾。
- **修法**：合并到最近的父章节（融入核心流程描述）。


### 问题 4：认知流差
章节顺序不符合读者的自然阅读路径，导致需要来回跳转才能理解。

**反例 A（章节夹心）**：
- **描述**："使用场景"夹在"需求概述"和"检查项"之间。
- **修法**：使用场景提到需求概述之后、检查项之前的最前端。

**反例 B（参考信息放中间）**：
- **描述**："timeframes 定义"（纯参考信息）放在中间打断阅读流。
- **修法**：纯参考信息移到文档末尾（附录位置）或实现要点中。

**反例 C（整体顺序混乱）**：
- **描述**：旧顺序为 需求 → 实现 → 使用场景混杂。
- **修法**：重构为 概述 → 使用方式 → 检查项 → 输出 → 实现要点 → 未来可扩展。

### 问题 5：过时内容

文档中保留了已废弃的功能/参数/别名。

**反例**：
- **描述**：文档提到 `--tf` 作为 `--timeframe` 的别名，但代码中已移除该别名。
- **修法**：grep 代码确认参数/功能是否仍存在，不存在则从文档中删除。

### 问题 6：缺失

文档该有但没有的内容——不是错了，是缺了。规整阶段只标注缺口（`（待补充：xxx）`），不现场补内容。

**反例 A（验证记录缺失）**：
- **描述**：各轮详情只记录了"做了什么"，没有验证结果（如"跑了一次15m确认152条缺口"），读者无法判断产出是否正确。
- **修法**：标注 `（待补充：验证记录）`，下一轮写作时补上。

**反例 B（应对落地状态缺失）**：
- **描述**：弯路列出了"应对"，但没标注哪些已落地、哪些只是原则。
- **修法**：标注 `（待补充：落地状态）`，下一轮写作时补上 `（已执行）` / `（原则）` 标签。

### 问题 7：粒度不对称

同级章节的详略程度不一致——有的有表格+教训+应对，有的只有干巴巴两行。

**反例**：
- **描述**：第1轮有详细动作表格+教训+应对，第5轮只有动作表格无教训；理想流程的 7 个步骤，有的步一行，有的步两行带注释。
- **修法**：同级章节采用同一种模板（如都带表格/都带教训），或者显式将不对称的章节降级/升级。

## 执行要点

- **只审查，不生成新内容**：规整阶段只做重组/合并/删除，不引入新概念或新方案。如发现缺失，标注 `（待补充）` 而非现场展开
- **先出问题清单再动手**：扫描完整文档后先列出问题表，用户确认后再逐项执行
- **对照代码**：涉及实现的文档，规整时必须对照源码确认示例/伪代码/数据结构是否同步
